## ADR-001: 文件即数据源

**日期**：2026-06-10
**状态**：✅ 已确认

### 背景

PlotFlow 需要存储用户的分支叙事数据。选项包括：SQLite 本地数据库（性能好，可索引）、IndexedDB（浏览器端存储）、或直接使用 `.mdstory` 文件。

### 决策

**`.mdstory` 文件是唯一数据源。绝不引入数据库存储脚本内容。**

### 后果

**正面**：
- 用户可以用任何文本编辑器打开/编辑文件
- Git 友好（纯文本 diff）
- 零同步冲突（文件即数据）
- 用户拥有数据主权

**负面**：
- 无法对内容做高效结构化查询（全文搜索需遍历目录）
- 大型项目（1000+ 节点）解析性能依赖解析器优化
- 没有版本历史（依赖用户自己的 Git）

**风险缓解**：
- M2 引入虚拟滚动应对大节点数
- 自动保存时保留 .bak 备份文件
- V0.2 评估是否需要 `.plotflow/index/` 元数据缓存

### 替代方案

| 方案 | 否决原因 |
|------|---------|
| SQLite 存储 .mdstory 内容 | 引入同步冲突，数据锁死在数据库，不可文本编辑器打开 |
| IndexedDB | Electron 环境下不必要，且与文件系统脱钩 |
| 文件+数据库双写 | 复杂度翻倍，一致性难以保证 |

## ADR-002: Electron 28 + React 18 桌面框架初始选型

**日期**：2026-06-10
**状态**：↩️ 已被 ADR-009 更新。历史决策保留用于解释项目初始框架选择，不再代表当前发行运行时版本。

### 背景

PlotFlow 需选择桌面应用框架。候选：Electron、Tauri、或纯 Web 应用（带本地文件 API）。

### 决策

**初始选择：Electron 28 + React 18 + TypeScript 5 (strict) + Vite 5。**

2026-06-24 之后，当前发行线更新为 Electron 42.5.0，详见 ADR-009。

### 后果

**正面**：
- Monaco Editor 官方 Electron 支持
- React Flow 原生 React 组件，无需适配层
- 10 年生态成熟，社区资源丰富
- pnpm workspace 天然支持 monorepo

**负面**：
- 安装包体积大（~150MB）
- 内存占用较高（~200MB 基准）
- Tauri 更轻量但生态不成熟

**风险缓解**：
- 使用 electron-vite 统一构建配置
- V0.1 接受安装包体积，后续继续评估 Tauri 迁移可行性
- Electron 主版本安全风险由 ADR-009 的受支持稳定线策略接管

### 替代方案

| 方案 | 否决原因 |
|------|---------|
| Tauri | Monaco Editor 在 Tauri webview 中未经充分验证；生态年轻 |
| 纯 Web | 无法访问本地文件系统、原生菜单、文件关联 |
| NW.js | 社区萎缩，生态不如 Electron |

## ADR-003: 解析器作为独立包 @plotflow/core

**日期**：2026-06-10
**状态**：✅ 已确认

### 背景

解析器、验证器、导出器这些纯 TypeScript 逻辑应该与 Electron/React UI 解耦，以便：
- 独立单元测试
- 未来发布为 npm 包供 CLI 工具使用
- Godot/Unity 插件通过命令行调用

### 决策

**解析器作为独立 pnpm workspace 包 `@plotflow/core`（后改名为 `@plotflow/parser`），零 UI 依赖。**

### 后果

**正面**：
- 核心逻辑可在 Node.js 环境独立运行
- 测试不依赖 Electron 或浏览器
- CI 可独立验证核心包

**负面**：
- monorepo 初期配置复杂度增加
- 类型定义需要在两个包之间共享

**风险缓解**：
- 使用 pnpm workspace protocol: `"@plotflow/core": "workspace:*"`
- 共享类型定义放在 `packages/core/src/types/`，app 通过导入引用

### 替代方案

| 方案 | 否决原因 |
|------|---------|
| 单包 monolith | 测试耦合，CLI 无法复用，违反分层原则 |
| 独立 npm 发布 | V0.1 阶段过于复杂，先 monorepo 后拆 |

## ADR-004: 补全引擎纯本地 N-gram

**日期**：2026-06-10
**状态**：✅ 已确认

### 背景

编辑器需要智能补全（节点标题、选项句式、正文描述、变量名）。选项包括：云端 LLM API、本地 LLM（如 Ollama）、或传统 N-gram 统计模型。

### 决策

**纯本地 N-gram 统计模型（1-5 gram），零网络依赖。**

### 后果

**正面**：
- 隐私安全，数据不离开本地
- 离线可用，无网络延迟
- 不需要 GPU/大内存

**负面**：
- 补全质量不如 LLM（尤其是长文本连贯性）
- 冷启动阶段补全质量差，依赖预置语料

**风险缓解**：
- 预置 5MB 精标注语料（RPG/视觉小说/解谜/通用分类）
- 本地增量学习（用户每次保存时后台训练）
- V0.3 后评估可选云补全插件

### 替代方案

| 方案 | 否决原因 |
|------|---------|
| 云端 LLM API | 强制联网，隐私风险，延迟不稳定 |
| 本地 Ollama | 要求用户安装额外服务，增加复杂度和硬件门槛 |
| 纯规则补全 | 补全范围受限，无法学习用户风格 |

## ADR-005: 亮色默认主题

**日期**：2026-06-11
**状态**：✅ 已确认

### 背景

编辑器需要默认主题。开发者工具常用暗色，但 PlotFlow 的目标用户是叙事设计师。

### 决策

**默认亮色主题。场景驱动选择：叙事设计师在下午三点工作室，自然光照充足，屏幕内容应与周围环境协调。暗色作为可切换选项。**

### 后果

**正面**：
- 符合叙事设计师的工作环境（通常白天工作）
- 与 PRD 定义的暖中性色调一致
- 亮色主题下语法高亮色值更易区分

**负面**：
- 部分开发者用户偏好暗色（可切换解决）

### 替代方案

| 方案 | 否决原因 |
|------|---------|
| 默认暗色 | 叙事设计师非程序员，暗色不符合工作环境 |
| 跟随系统 | 过于被动，"下午三点的工作室"场景下大概率是亮色 |

## ADR-006: shadcn/ui 组件库

**日期**：2026-06-11
**状态**：✅ 已确认

### 背景

编辑器需要统一的 UI 组件体系。选项包括：Ant Design、Material UI、shadcn/ui、或完全自研。

### 决策

**shadcn/ui（基于 Radix primitives + 自定义样式），语义多色策略。**

### 后果

**正面**：
- Radix 提供无障碍 primitives（键盘导航、ARIA）
- 样式完全可控（不锁死在第三方主题）
- 组件代码在项目中（非 node_modules 黑盒），可自由修改

**负面**：
- 组件数量不如 Ant Design / MUI 丰富
- 需要自行处理一些复杂交互（如条件编辑器）

### 替代方案

| 方案 | 否决原因 |
|------|---------|
| Ant Design | 样式与设计 Brief 偏离大，定制成本高 |
| Material UI | Google 风格过强，不适合工具类应用 |
| 完全自研 | V0.1 时间不允许，且无障碍需从零做起 |

## ADR-007: 分支图双模式（迷你地图 + 并排均衡）

**日期**：2026-06-11
**状态**：✅ 已确认

### 背景

分支图与编辑器的空间分配是核心 UX 决策。选项：仅迷你地图、仅并排、或双模式可切换。

### 决策

**双模式可切换，默认迷你地图（200×150 右下角），可展开为 40% 宽度的并排均衡模式。300ms 缩放动画过渡。**

### 后果

**正面**：
- 默认迷你地图不抢占编辑空间（叙事设计师主要工作是写文本）
- 需要时一键展开，获得完整可视化
- 不学 Articy — 避免臃肿的属性面板覆盖

**负面**：
- 两种模式需要维护两套布局状态
- 迷你地图下拖拽连线交互受限（需展开后方可编辑）

### 替代方案

| 方案 | 否决原因 |
|------|---------|
| 仅并排 | 叙事设计师主要写文本，固定 40% 空间浪费 |
| 仅迷你地图 | 无法编辑分支图，失去可视化编辑价值 |
| 对话框弹出 | 模态打断工作流，不可接受 |

## ADR-008: Graph-first Dual Projection

**日期**：2026-06-23
**状态**：✅ 已确认

### 背景

发行审计后，用户明确提出希望 PlotFlow 后续支持“完全基于流程图进行完全 GUI 操控”，并要求参考 UE5 蓝图及类似图表软件的操作逻辑。当前 V0.1 稳定模式是 split 分栏：Monaco 源文本为主，React Flow 图为辅助。这个模式对熟悉 Markdown 的用户有效，但无法满足不想直接写语法的叙事设计师。

同时，项目已有不可变约束：`.mdstory` 文件即数据源、Git 友好、离线优先、不得引入数据库或专有二进制故事格式。

### 决策

**新增 Graph Lab Core：流程图优先 GUI 工作区。`.mdstory` 仍是唯一磁盘真相源，split 源文本编辑与 Graph Lab GUI 编辑是同一故事数据的两个投影。**

M8 先交付图优先正式入口的核心用户旅程：
- 全屏 React Flow 画布作为主工作区。
- 左侧节点 palette 创建章节、节点、结局。
- 右侧 Inspector 编辑节点标题、章节、正文、选项、条件、效果和变量引用。
- 拖线可连接已有节点，也可拖到空白处创建目标节点并连接。
- 拖拽节点实时移动，松手后把坐标写入 `.mdstory` 的可选 `layout.graph.nodes` 块。
- 线缆拖到空白处显示动作菜单，支持创建节点并连接、搜索已有节点、断开既有连接或取消。
- Source Drawer 可折叠显示源文本，但不作为 Graph Lab 的主编辑面。
- 所有 GUI 操作必须序列化回 `.mdstory`，再通过现有 parse pipeline 同步图、诊断和导出。

### 后果

**正面**：
- 支持不想手写语法的用户完成核心创作闭环。
- 与 UE5 蓝图、Twine、Miro 类图软件的“画布为主”心智模型更接近。
- 保留 `.mdstory` 的文本可读、Git diff、外部编辑和引擎集成优势。
- Graph Lab 作为正式入口之一，不破坏当前 split 稳定工作流。

**负面**：
- 需要新增 `workspaceMode`、`graphEditService`、Inspector、Source Drawer、图模式 E2E 等一组跨模块能力。
- GUI 操作与 Monaco 撤销栈同步复杂，必须避免 React Flow store 直接写入导致双向同步竞态。
- 图模式会暴露更多 React Flow 渲染和 DOM 稳定性问题，E2E 需要保留真实 DOM 断言。

**风险缓解**：
- M8 只新增兼容旧文件的可选布局投影 `layout.graph.nodes`；它不改变剧情语义，缺失时继续使用自动布局。
- 所有图形化编辑先通过命令层生成可测试的故事编辑操作，再落回源文本。
- Graph Lab 不替代 split 默认工作流；两个入口共用同一 `.mdstory` 真相源。
- 每个 GUI 操作必须有“源文本结果 + 图 DOM 状态 + 导出结果”三层验收。

### 替代方案

| 方案 | 否决原因 |
|------|---------|
| 完全改为图数据库/二进制工程文件 | 违反文件即数据源、Git 友好和不锁死数据的核心承诺 |
| 保持文本主编辑不做图优先 | 无法满足用户提出的完全 GUI 操作目标 |
| GUI 只写 React Flow store，不回写 `.mdstory` | 会形成双数据源，导出、自动保存和外部编辑无法保持一致 |
| 立即替换 split 模式 | 风险过大，当前 split 已覆盖发行基础闭环，应以并行入口验证 |

## ADR-009: Electron 42 支持线安全迁移

**日期**：2026-06-24
**状态**：✅ 已确认

### 背景

发行审计发现 Electron 28 已不在官方最新三个稳定主版本支持窗口内。此前为了恢复发行门禁，项目曾临时保留 Electron 28 并用风险接受文档记录剩余 GHSA；这只能作为历史过渡方案，不能作为正式 Windows 安装包的安全放行依据。

### 决策

**桌面运行时迁移到 Electron 42.5.0，并移除 `pnpm.auditConfig.ignoreGhsas` 作为发行门禁路径。**

配套约束：
- `electron-vite`、Playwright Electron、`electron-builder` 必须与 Electron 42 一起通过全量门禁。
- `pnpm.cmd audit --audit-level moderate` 必须无已知漏洞；不再依赖 Electron 28 GHSA ignore。
- Windows 正式包必须由显式 `electron-builder.config.js` 构建，产物位于 `release/`，且不包含独立官网目录 `website/`。

### 后果

**正面**：
- 消除 Electron 28 安全风险接受对正式发行的阻断。
- Playwright E2E、菜单 IPC、文件打开、Graph Lab 和打包烟测在当前受支持 Electron 主版本线上验证。
- `pnpm audit` 恢复为真实安全门禁，不再静默忽略 Electron advisory。

**负面**：
- Electron 主版本升级可能带来渲染、菜单、preload、打包和 Playwright teardown 兼容问题，需要全量回归。
- 安装包体积和 Electron 下载缓存会随主版本迁移重新变化。

**风险缓解**：
- 全量运行 lint/typecheck/test/build/lint:css/E2E/audit。
- `pnpm.cmd package:win` 后执行 packaged smoke，覆盖命令行打开 `.mdstory`、Graph Lab GUI 编辑、Source Drawer 和 JSON 导出。
- 使用 asar 路径扫描确认 `website/` 未进入桌面应用包。

### 替代方案

| 方案 | 否决原因 |
|------|---------|
| 保留 Electron 28 + GHSA ignore | 不符合正式公开发行安全门禁，只能作为历史过渡方案 |
| 迁移到 40/41 | 可作为 Electron 42 兼容性失败时的 fallback；本轮 Electron 42.5.0 已通过 |
| 迁移到 Tauri | 超出本轮发行范围，Monaco/React Flow/文件关联/打包链路需重新验证 |

## ADR-010: 官方主题编译内置热插拔架构

**日期**：2026-06-25
**状态**：✅ 已确认

### 背景

Graph Lab 和官网视觉审查后确认，旧主题实现只覆盖 manifest、token、Monaco 和少量 layoutRecipe，核心视觉仍由 `StoryNodeCard`、`StoryEdge`、`branch-graph.css` 与通用面板组件决定。E2E 只验证功能可用和布局不遮挡，无法证明主题体验达标。

新产品方向是先发行官方主题，并让官方主题拥有足够深的视觉替换能力。当前不开放社区主题、本地 `.pf-theme.zip` 导入或远程主题下载，避免产品路线同时背负主题生态、安全、授权和支付复杂度。

### 决策

**主题系统采用官方编译内置模块热插拔架构。官方主题可以替换 token、Monaco 主题、layoutRecipe、motionRecipe、资产和受控 React slots；产品 UI 不暴露社区主题或本地导入入口。**

首版能力：
- 新增 `OfficialThemeDefinition`，包含 `manifest`、`tokens`、`monacoTheme`、`assets`、`layoutRecipe`、`motionRecipe`、`storeMeta` 和 `slots`。
- 首发官方主题为 `plotflow-narrative-workbench`（叙事工作台 / Narrative Workbench）和 `plotflow-blueprint-nightwatch`（夜航蓝图 / Blueprint Nightwatch）。
- Graph Lab 的节点、线缆、端口、Inspector、Source Dock、内容浏览器卡片等视觉由当前官方主题 slots 提供。
- Split 模式、ProblemPanel、ExportDialog、HomeSurface 和 ThemeCenter 消费同一主题 token 与主题 metadata。
- UI 提供 HomeSurface 主题模块、Topbar 主题入口、ThemeCenter 启用/重置默认、购买更多官方主题的外部官网/商店跳转。
- 旧 `ThemePackManifest` 和 `plotflow:themePack` 仅作为迁移兼容层，映射到官方主题 ID，不作为产品入口。

### 后果

**正面**：
- 官方主题能真正改变节点、线缆、端口和面板层级，不再只是背景和 token 变化。
- 主题切换可以在运行时热插拔生效，无需重启，也不改变 `.mdstory` 内容、导出 JSON 或 Graph Lab 命令层。
- 当前只发行官方主题，减少首版交付中的安全、生态、授权和兼容性风险。
- 后续如要开放市场，可先从官方商店、授权和下载链路独立推进，不影响当前编辑器发行。

**负面**：
- 新主题需要编译进应用，无法像社区主题包一样运行时下载导入。
- 官方主题 slots 增加组件边界复杂度，E2E 必须验证 renderer class、edge slot、CSS vars 和截图差异。
- 购买入口首版只能跳转外部官网/商店，不处理支付、授权、下载或解锁。

### 替代方案

| 方案 | 否决原因 |
|------|---------|
| 声明式 `.pf-theme.zip` 主题包 | 无法替换核心 React 节点/线缆，容易继续停留在 token 级皮肤；同时会引入导入、校验、资源限制和兼容性成本 |
| 任意 CSS/JS/React 社区插件 | 最大自定义能力足够强，但会破坏 Electron 安全边界和保存/导出闭环，不适合首版 |
| 只保留内置亮/暗主题 | 无法支撑“跨代工具感”和后续官方主题商业化 |
| 立即做联网内置市场 | 需要后端、授权、支付、签名、下载、审核和回退策略，超出本轮主题重设计范围 |
