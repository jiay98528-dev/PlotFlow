## ADR-001: 文件即数据源

**日期**：2026-06-10
**状态**：✅ 已确认

### 背景

PlotFlow 需要存储用户的分支叙事数据。选项包括：SQLite 本地数据库（性能好，可索引）、IndexedDB（浏览器端存储）、或直接使用 `.mdstory` 文件。

### 决策

**`.mdstory` 文件是唯一数据源。绝不引入数据库存储脚本内容。**

### 后果

**正面**：
- 用户可以用任何文本编辑器打开/编辑文件
- Git 友好（纯文本 diff）
- 零同步冲突（文件即数据）
- 用户拥有数据主权

**负面**：
- 无法对内容做高效结构化查询（全文搜索需遍历目录）
- 大型项目（1000+ 节点）解析性能依赖解析器优化
- 没有版本历史（依赖用户自己的 Git）

**风险缓解**：
- M2 引入虚拟滚动应对大节点数
- 自动保存时保留 .bak 备份文件
- V0.2 评估是否需要 `.plotflow/index/` 元数据缓存

### 替代方案

| 方案 | 否决原因 |
|------|---------|
| SQLite 存储 .mdstory 内容 | 引入同步冲突，数据锁死在数据库，不可文本编辑器打开 |
| IndexedDB | Electron 环境下不必要，且与文件系统脱钩 |
| 文件+数据库双写 | 复杂度翻倍，一致性难以保证 |

## ADR-002: Electron 28 + React 18 桌面框架

**日期**：2026-06-10
**状态**：✅ 已确认

### 背景

PlotFlow 需选择桌面应用框架。候选：Electron、Tauri、或纯 Web 应用（带本地文件 API）。

### 决策

**Electron 28 + React 18 + TypeScript 5 (strict) + Vite 5**

### 后果

**正面**：
- Monaco Editor 官方 Electron 支持
- React Flow 原生 React 组件，无需适配层
- 10 年生态成熟，社区资源丰富
- pnpm workspace 天然支持 monorepo

**负面**：
- 安装包体积大（~150MB）
- 内存占用较高（~200MB 基准）
- Tauri 更轻量但生态不成熟

**风险缓解**：
- 锁定 Electron 28.x，避免大版本 API 不稳定
- 使用 electron-vite 统一构建配置
- V0.1 接受安装包体积，V0.3 后评估 Tauri 迁移可行性

### 替代方案

| 方案 | 否决原因 |
|------|---------|
| Tauri | Monaco Editor 在 Tauri webview 中未经充分验证；生态年轻 |
| 纯 Web | 无法访问本地文件系统、原生菜单、文件关联 |
| NW.js | 社区萎缩，生态不如 Electron |

## ADR-003: 解析器作为独立包 @plotflow/core

**日期**：2026-06-10
**状态**：✅ 已确认

### 背景

解析器、验证器、导出器这些纯 TypeScript 逻辑应该与 Electron/React UI 解耦，以便：
- 独立单元测试
- 未来发布为 npm 包供 CLI 工具使用
- Godot/Unity 插件通过命令行调用

### 决策

**解析器作为独立 pnpm workspace 包 `@plotflow/core`（后改名为 `@plotflow/parser`），零 UI 依赖。**

### 后果

**正面**：
- 核心逻辑可在 Node.js 环境独立运行
- 测试不依赖 Electron 或浏览器
- CI 可独立验证核心包

**负面**：
- monorepo 初期配置复杂度增加
- 类型定义需要在两个包之间共享

**风险缓解**：
- 使用 pnpm workspace protocol: `"@plotflow/core": "workspace:*"`
- 共享类型定义放在 `packages/core/src/types/`，app 通过导入引用

### 替代方案

| 方案 | 否决原因 |
|------|---------|
| 单包 monolith | 测试耦合，CLI 无法复用，违反分层原则 |
| 独立 npm 发布 | V0.1 阶段过于复杂，先 monorepo 后拆 |

## ADR-004: 补全引擎纯本地 N-gram

**日期**：2026-06-10
**状态**：✅ 已确认

### 背景

编辑器需要智能补全（节点标题、选项句式、正文描述、变量名）。选项包括：云端 LLM API、本地 LLM（如 Ollama）、或传统 N-gram 统计模型。

### 决策

**纯本地 N-gram 统计模型（1-5 gram），零网络依赖。**

### 后果

**正面**：
- 隐私安全，数据不离开本地
- 离线可用，无网络延迟
- 不需要 GPU/大内存

**负面**：
- 补全质量不如 LLM（尤其是长文本连贯性）
- 冷启动阶段补全质量差，依赖预置语料

**风险缓解**：
- 预置 5MB 精标注语料（RPG/视觉小说/解谜/通用分类）
- 本地增量学习（用户每次保存时后台训练）
- V0.3 后评估可选云补全插件

### 替代方案

| 方案 | 否决原因 |
|------|---------|
| 云端 LLM API | 强制联网，隐私风险，延迟不稳定 |
| 本地 Ollama | 要求用户安装额外服务，增加复杂度和硬件门槛 |
| 纯规则补全 | 补全范围受限，无法学习用户风格 |

## ADR-005: 亮色默认主题

**日期**：2026-06-11
**状态**：✅ 已确认

### 背景

编辑器需要默认主题。开发者工具常用暗色，但 PlotFlow 的目标用户是叙事设计师。

### 决策

**默认亮色主题。场景驱动选择：叙事设计师在下午三点工作室，自然光照充足，屏幕内容应与周围环境协调。暗色作为可切换选项。**

### 后果

**正面**：
- 符合叙事设计师的工作环境（通常白天工作）
- 与 PRD 定义的暖中性色调一致
- 亮色主题下语法高亮色值更易区分

**负面**：
- 部分开发者用户偏好暗色（可切换解决）

### 替代方案

| 方案 | 否决原因 |
|------|---------|
| 默认暗色 | 叙事设计师非程序员，暗色不符合工作环境 |
| 跟随系统 | 过于被动，"下午三点的工作室"场景下大概率是亮色 |

## ADR-006: shadcn/ui 组件库

**日期**：2026-06-11
**状态**：✅ 已确认

### 背景

编辑器需要统一的 UI 组件体系。选项包括：Ant Design、Material UI、shadcn/ui、或完全自研。

### 决策

**shadcn/ui（基于 Radix primitives + 自定义样式），语义多色策略。**

### 后果

**正面**：
- Radix 提供无障碍 primitives（键盘导航、ARIA）
- 样式完全可控（不锁死在第三方主题）
- 组件代码在项目中（非 node_modules 黑盒），可自由修改

**负面**：
- 组件数量不如 Ant Design / MUI 丰富
- 需要自行处理一些复杂交互（如条件编辑器）

### 替代方案

| 方案 | 否决原因 |
|------|---------|
| Ant Design | 样式与设计 Brief 偏离大，定制成本高 |
| Material UI | Google 风格过强，不适合工具类应用 |
| 完全自研 | V0.1 时间不允许，且无障碍需从零做起 |

## ADR-007: 分支图双模式（迷你地图 + 并排均衡）

**日期**：2026-06-11
**状态**：✅ 已确认

### 背景

分支图与编辑器的空间分配是核心 UX 决策。选项：仅迷你地图、仅并排、或双模式可切换。

### 决策

**双模式可切换，默认迷你地图（200×150 右下角），可展开为 40% 宽度的并排均衡模式。300ms 缩放动画过渡。**

### 后果

**正面**：
- 默认迷你地图不抢占编辑空间（叙事设计师主要工作是写文本）
- 需要时一键展开，获得完整可视化
- 不学 Articy — 避免臃肿的属性面板覆盖

**负面**：
- 两种模式需要维护两套布局状态
- 迷你地图下拖拽连线交互受限（需展开后方可编辑）

### 替代方案

| 方案 | 否决原因 |
|------|---------|
| 仅并排 | 叙事设计师主要写文本，固定 40% 空间浪费 |
| 仅迷你地图 | 无法编辑分支图，失去可视化编辑价值 |
| 对话框弹出 | 模态打断工作流，不可接受 |
